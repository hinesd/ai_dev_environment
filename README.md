# ClawBot

Dockerized [OpenClaw](https://github.com/openclaw) gateway behind a **Caddy** reverse proxy, with local **Ollama** and cloud **DeepSeek** inference.

## Setup

### Prerequisites

- **Docker** (Engine or Desktop) on the gateway host
- GNU Make
- A static LAN IP for the host — configure it in `.env`

### 1. DNS & firewall

Create a free account at [DuckDNS](https://duckdns.org) and register a subdomain. Set its IPv4 to your LAN IP. Add your domain and token to `.env`.

Open ports 80 and 443 on your host so Caddy can receive HTTPS requests from devices on your LAN. Restrict access to your trusted subnet only:

```bash
sudo ufw allow from 10.20.30.0/24 to 10.20.30.5 port 80 proto tcp
sudo ufw allow from 10.20.30.0/24 to 10.20.30.5 port 443 proto tcp
```

On macOS, enable the built-in firewall via System Settings and ensure Docker Desktop is allowed to accept incoming connections.

### 2. Configure

All settings live in a single `.env` file at the project root (copy from `.env.example`):

```
# .env  (gitignored)
MODEL_DEFAULT=ollama/qwen2.5:7b
DEBUG=true
STATIC_IP=10.20.30.5
TRUSTED_SUBNET=10.20.30.0/24
DOCKER_SUBNET=172.30.0.0/16
GATEWAY_PORT=18789
OPENCLAW_GATEWAY_PASSWORD=<auto-generated>
DUCKDNS_DOMAIN=my-local-server.duckdns.org
DUCKDNS_TOKEN=<your-token>
DEEPSEEK_API_KEY=sk-...
```

### 3. Deploy

```bash
make init
```

Runs `verify_env → gen_password → down → build → ollama-model-pull → config → run → wait-healthy → doctor-lint`. When it returns, the system is running and healthy.

### 4. Pair your browser

Visit `https://<your DUCKDNS_DOMAIN>`. Find and approve the pairing request:

```bash
make devices-pending
```

Click **Reconnect** in the browser. Done once per device.

## LLM Backend

Controlled by `MODEL_DEFAULT` in `.env`:

```
# .env
MODEL_DEFAULT=ollama/qwen2.5:7b    # local inference
# MODEL_DEFAULT=deepseek/deepseek-chat  # cloud inference
DEBUG=true
```

| `MODEL_DEFAULT` | Provider | Runtime | Ollama container? |
|---|---|---|---|
| `ollama/qwen2.5:7b` (default) | Ollama | `qwen2.5:7b` in Docker | Yes |
| `deepseek/deepseek-chat` | DeepSeek | Cloud API | No |

```bash
make init                                           # ollama (default)
make init MODEL_DEFAULT=deepseek/deepseek-chat       # DeepSeek cloud
```

### Local inference (Ollama)

When `MODEL_DEFAULT` starts with `ollama/`, `make init` starts Ollama and pulls the model. Ollama listens on Docker DNS `ollama:11434`. Models are cached in the `ollama_data` Docker volume.

> **TODO**: Ollama's OpenAI-compatible endpoint rejects tool call arguments sent as objects (expects JSON strings). Tool-using turns fail with `json: cannot unmarshal object into Go struct field .messages.tool_calls.function.arguments`. Simple chat works fine.

```bash
make ollama-pull    # start Ollama + pull qwen2.5:7b
```

### Cloud inference (DeepSeek)

Set your API key in `.env`:

```bash
DEEPSEEK_API_KEY=sk-...
make restart MODEL_DEFAULT=deepseek/deepseek-chat
```

### Adding patches

Drop a JSON5 patch in `src/openclaw/patches/` and run `make config`. All `*.json5` files in the directory are applied automatically — each file targets a section of the openclaw config:

```
src/openclaw/patches/
├── gateway.json5              # gateway auth, control UI, trusted proxies
├── agents.json5               # agent defaults, model primary
├── models.ollama.json5        # ollama provider + tool restrictions
├── models.deepseek.json5      # deepseek provider
```

To add a new section (e.g. channels, session, tools), drop a corresponding `channels.json5` or `tools.json5` file. All patches are idempotent — they merge into the existing config without removing unrelated settings.

## Workspace

The agent workspace lives at `workspace/` (mounted to `~/.openclaw/workspace` inside the container). OpenClaw initializes it as a git repo on first run.

```bash
cd workspace && git remote add origin <your-private-repo-url> && git push -u origin main
```

## Daily Commands

```bash
make status         # container state + gateway health
make run            # start services
make down           # stop services (data preserved)
make restart        # stop → start → wait healthy
make config         # apply patches (first run or reapply)
make rebuild        # rebuild images from scratch + redeploy
make ollama-pull    # start Ollama + pull qwen2.5:7b
make cli ARGS="..."  # run any openclaw command
make doctor-lint    # read-only health check
make qr             # pairing QR for mobile devices
make reset          # full teardown and re-init
```

### Browser

2. Generate a QR code: `make qr`

## How it works

```
Ollama (Docker) ←─────────────────────────────┐
                                                │ Docker network
Caddy (Docker, TLS) ── 127.0.0.1 ──→ openclaw-gateway (Docker)
       ↑ shared network namespace               │
       └─────────────────────────────────────────┘
                          ↕
                    DeepSeek API (cloud)
                          ↑
                  LAN clients (10.20.30.0/24)
```

Caddy and the gateway share a network namespace (`network_mode: service:openclaw-gateway`). The gateway binds to `127.0.0.1` (loopback) and Caddy reaches it locally. The gateway port is published to host `127.0.0.1` only — no external direct access.

### Access control

Caddy limits which clients can reach the gateway using its `remote_ip` matcher. The allowed subnets are listed in `src/caddy/Caddyfile`:

```caddy
@trusted remote_ip \
    {env.TRUSTED_SUBNET} \
    {env.DOCKER_SUBNET}
```

Requests from any IP outside these subnets receive a `403 Forbidden`. The `TRUSTED_SUBNET` (your LAN, set in `.env`) allows devices on your local network. `DOCKER_SUBNET` (set in `.env`) allows traffic routed through Docker's bridge — on macOS, Docker Desktop rewrites incoming connection source IPs to the bridge IP, so this entry is required for LAN requests to pass Caddy's check.

To add a VPN or guest subnet, append another line:

```caddy
@trusted remote_ip \
    {env.TRUSTED_SUBNET} \
    {env.DOCKER_SUBNET} \
    192.168.1.0/24
```

Caddy terminates TLS at your DuckDNS domain via Let's Encrypt. Ollama runs on the Docker bridge network when `MODEL_DEFAULT` starts with `ollama/`.

Gateway auth uses `password` mode with a SecretRef (`{ source: "env", provider: "default", id: "OPENCLAW_GATEWAY_PASSWORD" }`) — the password never touches the config file. The Control UI prompts for the password on first visit.
