# ClawBot

Dockerized [OpenClaw](https://github.com/openclaw) gateway behind a **Caddy** reverse proxy, using **LM Studio** as the LLM backend.

## Architecture

```
┌──────────────┐     ┌──────────────────┐     ┌────────────┐     ┌──────────────┐
│  LM Studio   │◄────│ openclaw-gateway │◄────│   Caddy    │◄────│  LAN clients │
│  :1234       │     │ :18789           │     │ :80/:443  │     │  (browsers,  │
│  (host)      │     │ (Docker, no      │     │ (Docker,   │     │   phones)    │
└──────────────┘     │  host port)      │     │  host port)│     └──────────────┘
                     └──────────────────┘     └────────────┘
                          ▲                        │
                          │ 172.30.0.0/16          │ TLS (tls internal)
                          │ Docker bridge          │ X-Forwarded-User
                          └────────────────────────┘
                             trusted-proxy auth
```

| Component | Host | Notes |
|-----------|------|-------|
| **Caddy** | Docker (`openclaw-caddy`) | TLS termination, reverse proxy, injects `X-Forwarded-User` for trusted-proxy auth |
| **openclaw-gateway** | Docker (`openclaw-gateway`) | `ghcr.io/openclaw/openclaw:latest`, no host port exposure. All CLI commands run via `docker compose exec` inside this container. |
| **LM Studio** | Host | LLM backend, reached via `host.docker.internal:1234` |

Caddy binds to the host's LAN IP on ports 80/443 (not `0.0.0.0`). The gateway is only reachable through Caddy on the internal bridge network. This means:

- Browsers and mobile apps connect to `https://openclaw.home` → Caddy → Gateway
- The CLI runs inside the gateway container or network namespace (no external path needed)
- The gateway's Node.js port is never exposed to the LAN

## Prerequisites

- **Docker** (Engine or Desktop)
- **LM Studio** running on the host with `--bind 0.0.0.0`
- GNU Make
- A static LAN IP for the host (set as `OPENCLAW_LAN_IP` in `.env`)

## First-Time Setup

### 1. Configure local DNS

The domain `openclaw.home` needs to resolve to your host's LAN IP.

**UniFi**: Dashboard → Settings → Routing → Local DNS → add A-record `openclaw.home` → `10.20.30.5`

**Without UniFi**: add to `/etc/hosts` on each client device:
```
10.20.30.5  openclaw.home
```

### 2. Open host firewall

Allow ports 80/443 from your LAN subnet only (never from `0.0.0.0/0`):

```bash
sudo ufw allow from 10.20.30.0/24 to 10.20.30.5 port 80 proto tcp
sudo ufw allow from 10.20.30.0/24 to 10.20.30.5 port 443 proto tcp
```

### 3. Set your LAN IP in `.env`

Edit the `OPENCLAW_LAN_IP` variable in `.env` with your host's static LAN IP. Also verify `OPENCLAW_GATEWAY_PASSWORD` was auto-generated.

```bash
# .env
OPENCLAW_LAN_IP=10.20.30.5
OPENCLAW_GATEWAY_PASSWORD=<auto-generated>
```

### 4. Deploy

For a fresh deployment (or to tear down and rebuild):

```bash
make init
```

This runs: `verify_env → down → rebuild → up → wait-healthy → configure`. On first run it auto-generates `OPENCLAW_GATEWAY_PASSWORD` in `.env`.

For subsequent config changes without a full rebuild:

```bash
make setup
```

### 5. Trust the TLS certificate (optional but recommended)

Caddy generates a self-signed certificate. Browsers will show a warning until you import the root CA. On first visit click **Advanced → Proceed** to skip the warning, or install the CA permanently:

```bash
# Export the root CA from the Caddy container
docker compose exec -T caddy cat /data/caddy/pki/authorities/local/root.crt \
  | sudo tee /usr/local/share/ca-certificates/caddy-root.crt
sudo update-ca-certificates
```

For Android: copy the `root.crt` file to the device and install via **Settings → Security → Encryption & credentials → Install a certificate**.

### 6. Pair your first browser

Visit `https://openclaw.home`. The Control UI will show a **Device pairing required** prompt. On the host:

```bash
make devices-list                         # find the pending request ID
make devices-approve REQ=<request-id>     # approve it
```

Click **Reconnect** in the browser. You only need to do this once per browser/device.

## Daily Usage

```bash
make up          # start if stopped
make down        # stop
make restart     # down → up
make logs        # tail gateway logs
make setup       # rebuild + reconfigure
make reset       # full teardown and re-init
make configure   # re-apply seed.json (no rebuild)
make qr          # generate a pairing QR for mobile devices
```

### Pairing additional devices

Every new browser or mobile app needs one-time approval:

```bash
make qr                                    # (for phones) or visit https://openclaw.home (for browsers)
make devices-list                          # find the new pending request
make devices-approve REQ=<request-id>      # approve it
```

Paired devices reconnect automatically on subsequent visits/launches.

## Mobile Setup (Android)

1. Install [OpenClaw](https://play.google.com/store/apps/details?id=ai.openclaw.app) from Google Play
2. Run `make qr` on the host
3. In the app: **Connect → Setup Code → scan the QR**
4. The device auto-pairs (`autoApproveCidrs`). The app will show "checking node capability approval."
5. Approve the node's capabilities on the host:

```bash
make nodes-status                          # see "approval pending" or "reapproval pending"
make nodes-pending                         # prints copy-pasteable approval command
make nodes-approve REQ=<request-id>        # (or just copy the output from nodes-pending)
```

The QR encodes `wss://openclaw.home` with a short-lived bootstrap token. mDNS discovery is disabled (`OPENCLAW_DISABLE_BONJOUR=1`), so manual/QR setup is required for mobile. After the first pairing, the app auto-reconnects on launch.

> **Why two steps?** Starting with OpenClaw 2026.3.31, device pairing and node capability approval are separate. Device pairing (auto-approved via `autoApproveCidrs`) issues tokens. Node approval (manual) enables declared commands (camera, notifications, talk, etc.).

## Configuration

### seed.json (`config/seed.json`)

The single source of truth for gateway settings. Applied by `make configure` via `config set --batch-json`.

| Path | Value | Purpose |
|------|-------|---------|
| `gateway.mode` | `local` | Run as a local gateway |
| `gateway.bind` | `lan` | Listen on all interfaces (needed for Docker bridge) |
| `gateway.auth.mode` | `trusted-proxy` | Delegate auth to Caddy |
| `gateway.auth.trustedProxy.userHeader` | `X-Forwarded-User` | Header Caddy injects |
| `gateway.trustedProxies` | `["172.30.0.0/16"]` | Docker bridge subnet |
| `gateway.auth.password` | `${OPENCLAW_GATEWAY_PASSWORD}` | CLI fallback |
| `gateway.controlUi.allowedOrigins` | `["http://localhost:18789", "http://127.0.0.1:18789", "https://openclaw.home"]` | Allowed browser origins |
| `gateway.controlUi.allowInsecureAuth` | `false` | Require auth (Caddy provides it) |
| `agents.defaults.model.primary` | `lmstudio/openai/gpt-oss-20b` | Default LLM |
| `models.providers.lmstudio` | `{baseUrl, api, ...}` | LM Studio connection |

### Caddyfile (`config/Caddyfile`)

```caddyfile
openclaw.home {
    tls internal { on_demand }
    reverse_proxy openclaw-gateway:18789 {
        header_up X-Forwarded-User openclaw-operator
        header_up X-Forwarded-For {remote_host}
        header_up X-Forwarded-Proto {scheme}
        header_up X-Real-IP {remote_host}
    }
}
```

- `tls internal` — auto-generates self-signed certs
- `openclaw-gateway:18789` — resolved via Docker's internal DNS on the bridge network
- `X-Forwarded-User: openclaw-operator` — the identity header the gateway trusts

### Environment (`.env`)

| Variable | Purpose |
|----------|---------|
| `OPENCLAW_GATEWAY_PASSWORD` | Generated by `make init`. Used by CLI for device pairing and config changes. |
| `OPENCLAW_GATEWAY_TOKEN` | Disabled (commented out). Token auth conflicts with trusted-proxy mode. |
| `OPENCLAW_LAN_IP` | Host's static LAN IP. Caddy binds ports 80/443 to this interface. |

`.env` is gitignored. `.env.example` is the committed template.

## Authentication Model

```
Browser → Caddy (TLS) → Gateway
         │
         injects: X-Forwarded-User: openclaw-operator
         source IP: 172.30.0.3
                        │
                        ▼
         Gateway checks: is 172.30.0.3 ∈ trustedProxies [172.30.0.0/16]?
         → Yes → trusts the identity → session authenticated
```

**Trusted-proxy** means the gateway never handles credentials for browser/mobile clients. It only checks: "did this connection come from an IP I trust (Caddy)? If so, I trust the `X-Forwarded-User` header it stamped."

**Password fallback** is used only for local CLI operations (`make devices-approve`, `make qr`) where there's no Caddy proxy in front. These commands connect via loopback inside the container.

## Persistence

| Host Path | Container Path | Purpose |
|-----------|---------------|---------|
| `docker_persist/config/` | `/home/node/.openclaw/` | Gateway config (`openclaw.json`) |
| `docker_persist/workspace/` | `/home/node/.openclaw/workspace/` | Agent workspace |
| `docker_persist/auth-secrets/` | `/home/node/.config/openclaw/` | Encrypted auth secrets |
| `caddy_data` (named volume) | `/data/` | Caddy TLS certs and PKI |

These directories are gitignored. `make down` preserves volumes; `make clean` destroys them (new certs regenerate on next `up`).

The Makefile includes targets from three files under `.make/` plus composed targets in the top-level `Makefile`:

|--------|-------------|

| Target | Description |
|--------|-------------|
| `make build` | Build the gateway image |
| `make rebuild` | Rebuild from scratch (no cache) |
| `make up` | Start gateway + Caddy in detached mode |

| `make clean` | `down -v` (destroys volumes including Caddy certs) |
| `make logs` | Tail gateway logs |

### `.make/openclaw.mk`

| Target | Description |
|--------|-------------|
| `make configure` | Apply `seed.json` + restart gateway |
| `make qr` | Generate pairing QR for mobile |
| `make devices-list` | List pending and paired devices |
| `make devices-approve REQ=<id>` | Approve a pending device |
| `make nodes-status` | List connected nodes and their approval state |
| `make nodes-pending` | Print pending approval request IDs as copy-pasteable commands |
| `make nodes-approve REQ=<id>` | Approve node capabilities |
| `make onboarding` | Run the interactive onboarding wizard |
| `make security-audit` | Run OpenClaw security audit |
| `make cli CMD="..."` | Run an arbitrary CLI command |
| `make channels CMD="..."` | Manage messaging channels |
| `make dashboard` | Launch the dashboard inside the gateway container |

| Target | Composition |

## Troubleshooting

**Gateway fails to start with "trusted-proxy and token auth are mutually exclusive"**
The `OPENCLAW_GATEWAY_TOKEN` env var is set (even if empty). Comment it out in `.env` and restart.

**"pairing required: device is not approved" in logs**
A new browser/phone connected but hasn't been approved yet. Run `make devices-list` then `make devices-approve REQ=<id>`.

**"connection refused" in Caddy logs after restart**
The gateway container was still restart-looping. Caddy retries automatically once the gateway is healthy.

**Subnet mismatch after `docker compose down`**
The `openclaw-net` bridge is pinned to `172.30.0.0/16` in `docker-compose.yml`. Run `make restart` to recreate it with the correct subnet. Verify with `docker network inspect clawbot_openclaw-net | grep Subnet`.

## Notes

- **LM Studio networking**: The compose file maps `host.docker.internal` to the host gateway. LM Studio must listen on `0.0.0.0`.
- **No sandbox**: Container sandboxing is disabled. `openclaw/sandbox.Dockerfile` is retained for future use.
- **Docker Desktop**: Docker Desktop's Dev Environments GUI may inject `USER root` or `docker-ce-cli` lines into compose files. Revert with `git checkout -- openclaw/Dockerfile docker-compose.yml && make build`.
- **Bonjour/mDNS**: Disabled (`OPENCLAW_DISABLE_BONJOUR=1`). Mobile devices must use QR or manual setup — they won't auto-discover the gateway on the LAN.
